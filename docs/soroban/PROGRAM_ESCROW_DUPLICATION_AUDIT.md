# Duplication Audit: `program-escrow/src/lib.rs` vs `soroban/contracts/program-escrow/src/lib.rs`

**Date:** 2026-07-16
**Scope:** Documentation-only audit. No contract logic was changed.

## Verdict

`soroban/contracts/program-escrow/src/lib.rs` is **not a stale duplicate and not a
fork** of the real program-escrow contract. It is the **unmodified hello-world
placeholder** generated by `stellar contract init` when the `soroban/` workspace was
scaffolded (see [soroban/README.md — step 3](README.md#3-initialize-soroban-workspace),
which explicitly notes the init "creates initial workspace with hello-world contract
(can remove or replace later)").

There is **no shared contract logic between the two files**, so there is nothing to
sync and no risk of a fixed vulnerability being silently reintroduced through this
pair. The same applies to the sibling `soroban/contracts/escrow/src/lib.rs`, which is
an identical placeholder.

## Diff summary

| | `program-escrow/src/lib.rs` | `soroban/contracts/program-escrow/src/lib.rs` |
|---|---|---|
| Lines | 3,623 | 23 |
| Contents | Full ProgramEscrow contract: RBAC, whitelist, milestone schedules, disbursement, fees, pause/circuit-breaker, reentrancy guard, analytics events, dispute resolution, TTL management | Single `hello(env, to)` function returning `["Hello", to]` — the stock `stellar contract init` template, comments included verbatim |
| Storage usage | Instance + persistent storage with explicit TTL extension | None |
| Test suite | 20+ test modules under `program-escrow/src/` | One 21-line template test (`src/test.rs`) |
| Git history | Actively developed (most recently `29c8480`, persistent-TTL fix, PR #164) | Untouched since the initial commit (`8f4075d`); `git log --follow` shows exactly one commit |

A conventional line-by-line diff is not meaningful: the files have no common ancestry
beyond both being Soroban contracts. Every line of the real contract is "added"
relative to the placeholder, and the placeholder's 23 lines match the upstream
`stellar contract init` template byte-for-byte apart from the crate name.

## TTL-extension parity

The persistent-storage TTL fix merged for program-escrow
(`29c8480` — "feat(program-escrow): extend persistent TTL for schedules, history, and
program data", PR #164) introduced `bump_persistent_symbol_ttl`,
`bump_persistent_datakey_ttl`, and `bump_instance_ttl` helpers in
`program-escrow/src/lib.rs` (around lines 720–737) and wired them into the write
paths for schedules, history, and program data.

**This fix does not apply to the `soroban/` copy and is not missing from it.** The
placeholder performs no storage reads or writes of any kind (`extend_ttl` does not
appear in the file, and neither does any `env.storage()` call), so there is no
persistent state whose TTL could lapse. The same holds for the milestone-validation
fixes: the placeholder has no milestone logic.

## Is the `soroban/` copy live?

Partially, but only as a **toolchain smoke test**, not as deployable business logic:

- The CI job `soroban-contract-matrix` (`.github/workflows/ci.yml`) builds the
  `soroban/` workspace via `scripts/run_contract_matrix.sh` across pinned
  `soroban-sdk` versions and performs a deploy (or dry-run deploy) of the resulting
  wasm to a local sandbox/testnet. The script defaults to the `escrow` placeholder;
  `program-escrow` can be selected with `--contract program-escrow`.
- Nothing in `deployments/`, `scripts/`, or the SDK references the placeholder's
  `hello` entrypoint as a product surface.

So the crate is compiled and deployed by CI to validate the Stellar CLI / SDK matrix,
but it carries no escrow behavior. Its value is as a minimal, dependency-light wasm
target for the matrix job.

## Recommendation (out of scope for this audit)

To prevent recurring confusion, a follow-up issue could either:

1. Rename the `soroban/contracts/*` crates to something unambiguous
   (e.g. `matrix-smoke-test`) so they can never be mistaken for the production
   `program-escrow` / escrow crates, or
2. Point the CI matrix at the real contracts and delete the placeholders.

Either option should be handled separately, since the CI matrix job's caching and
SDK pinning depend on the `soroban/` workspace layout.
